## 3. 项目整体架构

### 3.1 架构总览

从代码结构上看，这个项目可以分成 5 层：

```text
VTK 文件输入层
-> 内部数据转换层
-> 核心算法层
-> 门面封装层
-> GUI / 测试程序层
```

更具体一点，就是：

```text
.vtk 文件
-> vtkDataSet
-> VTKDataConverter
-> DataObject
-> CAEProcessingFacade
-> GLGradientEngine / GLFilterEngine
-> 结果回写 DataObject
-> GUI 显示 / 导出 .vtk / 测试程序评估
```

### 3.2 各层的职责

#### 1. 输入层

负责把 `.vtk` 文件读成 VTK 官方的数据对象。

关键位置：

1. `CAEProcessingFacade::loadDatasetFromVTKFile(...)`
2. `vtkDataSetReader`

#### 2. 数据转换层

负责把 `vtkDataSet` 转成项目内部统一的数据结构 `DataObject`。

关键位置：

1. `VTKDataConverter.h`
2. `VTKDataConverter.cpp`

#### 3. 核心算法层

负责真正的 GPU 计算。

分成两大引擎：

1. `GLGradientEngine`
   负责梯度。
2. `GLFilterEngine`
   负责双边滤波和多尺度融合。

#### 4. 门面层

`CAEProcessingFacade` 是整个项目最重要的总入口。

它对外统一提供：

1. 加载数据集
2. 查询字段
3. 计算梯度
4. 做数据优化
5. 导出结果

#### 5. 表现层

分成两类：

1. GUI：`app/MainWindow.cpp`
2. 测试程序：`TestGradient.cpp`、`TestMultiScale.cpp`

如果老师问你“这个系统主要看哪些代码”，你可以优先回答下面这几个：

## 5. 内部数据模型：为什么需要 `DataObject`

### 5.1 为什么不能直接在 VTK 对象上做所有事

VTK 的数据结构很强大，但对于你的项目来说有两个问题：

1. 数据组织更偏通用可视化框架，不一定最适合直接送进 GPU。
2. 不同网格类型在 VTK 里差异较大，不利于你做统一的算法接口。

所以项目在中间增加了一层自己的内部表示：`DataObject`。

### 5.2 `DataObject` 里存了什么

`DataObject.h` 里最重要的字段包括：

1. `points`
   存点坐标，格式是 `[x0,y0,z0,x1,y1,z1,...]`
2. `cellCenters`
   存单元中心坐标
3. `dataArrays`
   存字段数组，包括点数据和单元数据
4. `pointNeighbors` + `pointNeighborOffsets`
   点邻域图
5. `cellNeighbors` + `cellNeighborsOffsets`
   单元邻域图
6. `cells`、`cellTypes`、`cellOffsets`
   单元连接关系
7. `dimensions`
   规则网格的尺寸

### 5.3 为什么邻域用 CSR 形式

点邻域和单元邻域都用了类似 CSR 的压缩邻接表示：

```text
neighbors[offsets[i] ... offsets[i+1]-1]
```

好处有 4 个：

1. 存储紧凑。
2. 容易顺序访问。
3. 很适合上传到 SSBO。
4. 规则网格和非结构网格都能统一用这种形式表达“邻域”。

### 5.4 一个非常重要但容易忽略的事实

当前系统内部的几何坐标和字段数组，基本都统一成了 `float` 存储。

比如：

1. `DataArray::data` 是 `std::vector<float>`
2. `points` 是 `std::vector<float>`
3. `cellCenters` 是 `std::vector<float>`

这对 GPU 友好，但也带来一个现实问题：

> 对于量级很大的真实工程场，或者几何尺度差异很大的薄体/高纵横比网格，`float` 会放大数值不稳定。

这也是后面你分析 `notch_stress` 这类困难样例时必须提到的一个技术点。

## 6. 数据进入系统以后发生了什么

### 6.1 加载入口

数据加载的总入口是：

```cpp
CAEProcessingFacade::loadDatasetFromVTKFile(...)
```

它的主要流程是：

```text
读取 .vtk
-> 得到 vtkDataSet
-> 创建 DatasetRecord
-> 调用 VTKDataConverter 转换
-> 可选地注入解析 benchmark 数组
-> 存入 m_records
```

这里有一个很重要的设计：

1. GUI 正常流程默认不注入解析 benchmark。
2. 解析 benchmark 只在测试程序里按需开启。

这符合你现在论文口径：解析场用于验证，不属于 GUI 日常使用流程。

### 6.2 `DatasetRecord` 是什么

每个数据集加载后，会放进 `CAEProcessingFacade` 内部的 `DatasetRecord`。

它包含：

1. 数据集 ID
2. 显示名
3. `DataObject`
4. 原始 `vtkDataSet`
5. 梯度结果列表
6. 点支撑缓存
7. 单元支撑缓存

也就是说：

> 一个数据集不是只存原始数据，还会存它的计算结果和中间支撑信息。

这样做的好处是：

1. 可以避免重复构造支撑。
2. 可以在 GUI 中不断追加新结果数组。
3. 可以导出时把所有结果一起写回 `.vtk`。

## 7. VTK 数据转换模块详解

### 7.1 为什么要做转换

项目里的很多计算并不直接面向 VTK 单元对象，而是面向：

1. 扁平坐标数组
2. 邻域图
3. 字段数组
4. 单元中心

所以 `VTKDataConverter` 的本质作用是：

> 把 VTK 的对象式数据组织转换为内部数组式数据组织。

### 7.2 总转换流程

入口函数：

```cpp
VTKDataConverter::convertVTKToInternal()
```

它会先判断网格类型，然后走两条分支：

1. 规则网格：`convertRegularGrid()`
2. 非结构网格：`convertUnstructuredGrid()`

### 7.3 规则网格转换

规则网格分支主要做：

1. `convertPoints()`
2. `convertDimensions()`
3. `convertDataArrays()`
4. `convertCellCenters()`
5. `convertCell()`

对应思路是：

1. 提取所有点坐标
2. 保存规则网格尺寸
3. 提取点数据和单元数据
4. 求出单元中心
5. 保存单元连接

### 7.4 非结构网格转换

非结构网格分支主要做：

1. `convertPoints()`
2. `convertDataArrays()`
3. `convertPointInCellNeighbors()`
4. `convertPointNeighborsRobust(12, 24)`
5. `convertCellCenters()`
6. `convertCell()`
7. `convertCellNeighbors()`

这说明非结构网格不仅要提取几何和字段，还要重点建立邻域图。

### 7.5 点邻域是怎么构造的

`convertPointNeighborsRobust(int minK, int knnK)` 的逻辑很值得你讲：

1. 先通过 `vtkStaticCellLinks` 找每个点所在的 incident cells。
2. 再从这些单元里收集拓扑相邻点。
3. 如果邻域点太少，就用 `vtkKdTreePointLocator` 做 KNN 补点。
4. 补点时不是无限乱补，而是限制在一个基于局部平均距离的可接受范围内。

这说明当前点邻域不是“纯 KNN”，也不是“纯拓扑邻域”，而是：

> 先拓扑，后几何补充的混合邻域策略。

这对于非结构网格特别重要，因为：

1. 纯拓扑邻域可能太少。
2. 纯 KNN 容易跨层、跨面、跨厚度连错。

### 7.6 单元邻域怎么构造

`convertCellNeighbors()` 的思路是拓扑优先。

它会根据单元维度决定怎么找邻居：

1. 3D 单元通过共享面找邻居
2. 2D 单元通过共享边找邻居
3. 1D 单元通过共享端点找邻居

如果这些都不方便，再退回到共享点的更宽松邻接。

这说明单元邻域更强调“拓扑相邻”，而不是纯几何距离。

### 7.7 单元中心怎么求

`convertCellCenters()` 的实现用了：

1. `cell->GetParametricCenter(...)`
2. `cell->EvaluateLocation(...)`

也就是说，单元中心不是简单把所有节点平均，而是通过 VTK 的单元参数坐标接口求出更稳妥的中心位置。

### 7.8 导出过程

导出时走：

```cpp
VTKDataConverter::convertInternalToVTK()
```

它会：

1. 重建点坐标
2. 重建结构网格或非结构网格
3. 把内部新增数组一并写回

这就是为什么你在 GUI 里算完梯度或者优化后，导出的 `.vtk` 里可以带着新数组一起保存。

## 8. 梯度模块：项目最核心的部分

这是老师最可能重点问你的部分。

### 8.1 梯度模块解决什么问题

在 CAE 后处理中，梯度可以用于：

1. 应力、温度、位移变化率分析
2. 边界或特征识别
3. 颜色映射增强
4. 后续等值面、粒子、流线等可视化模块的辅助量

所以梯度模块本质上是在回答：

> 给定一个场变量，如何在网格上稳定、高效地重构它的空间变化率。

### 8.2 梯度模块对外入口

统一入口是：

```cpp
CAEProcessingFacade::computeGradient(...)
```

这也是 GUI 按下 “Compute Gradient” 按钮后最终调用的接口。

GUI 的调用链是：

```text
MainWindow::computeGradient()
-> CAEProcessingFacade::computeGradient()
-> computeByFD() 或 computeByAdaptiveWLS()
-> GLGradientEngine
-> 对应的 compute shader
```

### 8.3 为什么需要 `Auto`

`CAEGradientRequest` 里允许你传：

1. `Auto`
2. `FiniteDifference`
3. `AdaptiveWeightedLeastSquares`

但当前真正稳定的公开策略是：

1. 规则网格自动走 `FD`
2. 非结构网格自动走 `AWLS`

对应代码就在：

```cpp
if (method == Auto) {
    if (gridType == Regular) method = FiniteDifference;
    else method = AdaptiveWeightedLeastSquares;
}
```

这个逻辑很重要，因为它体现了系统设计原则：

> 不是所有数据都强行走同一算法，而是根据网格性质自动选择更合适的主方法。

## 9. 规则网格梯度：`FD`

### 9.1 理论思想

规则网格最适合用有限差分。

对于标量场 $\phi$，内部点上的一维中心差分形式为：

$$
\frac{\partial \phi}{\partial x}
\approx
\frac{\phi(x+h)-\phi(x-h)}{2h}
$$

边界点则用单边差分。

扩展到三维后，就是分别求：

$$
\frac{\partial \phi}{\partial x}, \quad
\frac{\partial \phi}{\partial y}, \quad
\frac{\partial \phi}{\partial z}
$$

### 9.2 为什么这里不是最简单的笛卡尔差分

`FD.glsl` 里并不是直接写“除以固定步长”的最粗糙版本，而是还考虑了点坐标本身，构造了局部 Jacobian 和逆 Jacobian。

这说明当前规则网格分支的思路是：

1. 先在参数方向上做差分
2. 再通过 Jacobian 映射到物理空间

因此它不仅能处理严格均匀的 `vtkImageData`，也更适合处理一般结构网格或非等距结构情况。

### 9.3 代码流程

CPU 侧：

1. `computeGradient(...)`
2. `computeByFD(...)`
3. `GLGradientEngine::computeRegularFD(...)`

GPU 侧：

1. `FD.glsl`

### 9.4 `computeByFD(...)` 做了什么

它会：

1. 判断输入是点数据还是单元数据
2. 为点数据直接使用 `data.points`
3. 为单元数据改用 `data.cellCenters`
4. 组织 `RegularParams`
5. 调用 `m_engine.computeRegularFD(...)`

这里有一个你汇报时可以讲的细节：

> 当前规则网格不仅支持点数据梯度，也支持单元数据梯度。对于单元数据，系统把单元中心当成样本位置，再按规则网格尺寸做有限差分。

### 9.5 `FD.glsl` 的核心思想

它主要做 3 件事：

1. 根据当前点位置选择前后相邻样本
2. 计算参数方向上的差分
3. 用 Jacobian 逆矩阵把参数导数转成物理空间导数

它的优点是：

1. 对规则网格非常高效
2. GPU 并行友好
3. 思路清晰

它的局限是：

1. 只适合规则或结构化布局
2. 不适合一般非结构网格

## 10. 非结构网格梯度：`AWLS`

这部分是当前项目最能体现“算法设计”的地方。

### 10.1 为什么非结构网格不能直接用简单差分

非结构网格的问题在于：

1. 邻居数量不固定
2. 邻居分布不均
3. 单元形状和尺度差异大
4. 局部可能像线、像面、像体

所以不能像规则网格那样简单选“前后左右上下”做差分。

更合适的方式是：

> 在每个样本点的局部邻域上，建立一个局部线性模型，再通过最小二乘估计梯度。

### 10.2 局部线性近似

设中心点为 $\mathbf{x}_i$，邻域点为 $\mathbf{x}_j$，场值为 $\phi$，则可做局部一阶近似：

$$
\phi(\mathbf{x}_j) - \phi(\mathbf{x}_i)
\approx
\nabla \phi_i \cdot (\mathbf{x}_j - \mathbf{x}_i)
$$

把很多邻居都放在一起后，就可以通过最小二乘求一个最符合这些关系的梯度向量。

### 10.3 为什么是“加权”最小二乘

并不是所有邻居都应同等重要。

通常更近的邻居更可信，所以可以引入权重：

$$
\min_{\mathbf{g}}
\sum_j w_j
\left(
\phi_j - \phi_i - \mathbf{g}\cdot(\mathbf{x}_j-\mathbf{x}_i)
\right)^2
$$

其中：

1. 距离近的点权重大
2. 距离远的点权重小

这就是 WLS 的基本思想。

### 10.4 当前 AWLS 不是简单 WLS，而是“自适应支撑 + 局部维数识别 + 自适应正则”

你现在这版实现比普通 WLS 多了三层工程增强：

1. 邻域不是固定的，而是自适应构造
2. 局部几何维数会自动识别
3. 邻域质量不好时会自动增强正则

所以准确说法是：

> 当前非结构网格梯度模块采用基于局部支撑的自适应加权最小二乘梯度重构方法。

## 11. AWLS 的完整执行流程

### 11.1 第一步：算法入口

入口是：

```cpp
CAEProcessingFacade::computeByAdaptiveWLS(...)
```

它会先做两件事：

1. 检查当前确实是非结构网格
2. 调用 `ensureAdaptiveSupport(...)` 构造或复用支撑

### 11.2 第二步：构造自适应支撑

这一步是 AWLS 最关键的前处理。

核心函数是：

```cpp
ensureAdaptiveSupport(...)
-> buildAdaptiveGradientSupport(...)
```

支撑里包含：

1. `offsets`
2. `neighbors`
3. `frames`
4. `dimTags`
5. `quality`
6. `meanNeighborDistance`

这 6 类信息一起定义了后面 GPU 求解的“局部环境”。

### 11.3 第三步：候选邻域构造

每个点最开始会从基础邻接出发。

当前策略是：

1. 先收集一环拓扑邻域
2. 如果邻域太小，扩二环
3. 如果还是不够，再用 KNN 补点

这说明系统并没有盲目依赖某一种邻域来源，而是在尽量平衡：

1. 拓扑相关性
2. 几何完整性
3. 邻域数量下限

### 11.4 第四步：局部谱分析

有了候选邻域之后，系统会对局部几何做协方差谱分析。

其目的有 3 个：

1. 判断局部更像线、面还是体
2. 提取主方向框架 `frame`
3. 评估邻域质量 `quality`

### 11.5 第五步：局部维数识别

系统通过特征值比例来判断局部维度。

当前核心逻辑是：

1. 若 `r21 < lineEigenRatio`，判为 `dim = 1`
2. 否则若 `r31 < planeEigenRatio`，判为 `dim = 2`
3. 否则判为 `dim = 3`

其中：

1. `dimTag = 1` 表示局部更像线状
2. `dimTag = 2` 表示局部更像面状
3. `dimTag = 3` 表示局部更像体状

这一步特别重要，因为它决定了 GPU 端后续到底做 1D、2D 还是 3D 拟合。

### 11.6 第六步：质量评估

当前 `quality` 不是一个随便拍脑袋的数，而是由多项因素组合而成：

1. `spectralQuality`
2. `directionalQuality`
3. `countQuality`

也就是说，质量不是只看邻居数量，而是同时看：

1. 几何谱特征是否稳定
2. 方向分布是否平衡
3. 邻居数是否足够

### 11.7 第七步：邻域平衡化

系统还会对邻居进行方向分桶和平衡选择。

其目的不是花哨，而是很实际：

> 减少邻域都堆在一边导致的最小二乘病态和偏置。

### 11.8 第八步：把支撑信息送进 GPU

构造好的支撑会通过 `GLGradientEngine::computeUnstructuredAdaptiveWLS(...)` 上传到 GPU。

上传内容包括：

1. 样本坐标
2. 邻域偏移
3. 邻居索引
4. 场值数组
5. 局部框架 `frame`
6. 维数标签 `dimTag`
7. 邻域质量 `quality`
8. 平均邻距 `meanNeighborDistance`

### 11.9 第九步：GPU 求解

真正求解发生在：

```glsl
Shaders/AdaptiveWLS.glsl
```

它不是一个统一的三维拟合，而是根据 `dimTag` 分支处理：

1. `dim == 1`
   在主方向 `e1` 上做 1D 拟合
2. `dim == 2`
   在切平面 `e1, e2` 上做 2D 拟合
3. `dim == 3`
   做完整 3D 拟合

这就是 AWLS 名字里 “Adaptive” 的真正含义。

### 11.10 第十步：自适应正则

在 `AdaptiveWLS.glsl` 里，正则强度会随着质量变化。

质量越差，正则越强：

```text
lambdaScale = 1 + uLambdaAmplify * (1 - q)
```

这一步的目的很明确：

> 邻域越不可靠，就越要抑制病态求解。

### 11.11 第十一步：失败回退

如果 `AdaptiveWLS` 失败，系统还会退回到普通 `WLS`：

```cpp
m_engine.computeUnstructuredWLS(...)
```

因此当前结构实际上是：

```text
优先 AdaptiveWLS
失败时 fallback 到普通 WLS
```

这是一种工程上比较稳妥的设计。

## 12. 普通 `WLS` 和 `AWLS` 的关系

`WLS.glsl` 是更基础的版本。

它做的是标准三维加权最小二乘：

1. 计算邻域点相对中心点的位移向量
2. 构造加权法方程
3. 加正则项
4. 通过 Cholesky 分解求解梯度

这个版本不考虑：

1. 局部维数自适应
2. 局部框架
3. 邻域质量
4. 自适应正则增强

所以可以把两者关系理解为：

1. `WLS` 是基础版
2. `AWLS` 是工程增强版

## 14. 数据优化模块详解

这是项目的第二条主线。

### 14.1 它解决什么问题

真实仿真场往往会有：

1. 噪声
2. 局部毛刺
3. 尺度混叠
4. 细节与整体趋势难以兼顾

所以系统不能只算梯度，还需要对原始场做预处理。

当前数据优化模块的目标是：

> 通过图双边滤波和多尺度分解，得到更平滑、但又尽量保留特征的结果场。

### 14.2 对外入口

统一入口是：

```cpp
CAEProcessingFacade::computeMultiScaleDecompositionAndFusion(...)
```

GUI 中 “Run MultiScale Optimization” 按钮按下后，最终走的就是这个接口。

### 14.3 整体流程

数据优化模块的总体流程是：

```text
输入原始场
-> 构造图邻域
-> 估计空间尺度和数值尺度
-> 多层图双边滤波
-> 相邻平滑层相减得到细节层
-> 按权重融合细节
-> 输出 base / detail / fused
```

### 14.4 第一阶段：构造图

它会调用内部的 `buildFilterGraph(...)`。

规则网格上：

1. 自动生成六邻域

非结构网格上：

1. 点数据用点邻域
2. 单元数据用单元邻域

所以数据优化模块和梯度模块共享了“图邻域”这一核心思想。

### 14.5 第二阶段：估计尺度

系统会估计两个基础量：

1. `meanSpacing`
   邻域平均几何距离
2. `valueStd`
   字段值标准差

然后根据请求参数生成：

1. `spatialSigmaBase`
2. `rangeSigmaBase`
3. `edgeSigma`

这说明当前模块不是硬编码某个固定滤波尺度，而是让尺度跟数据本身相关。

### 14.6 第三阶段：图双边滤波

`GLFilterEngine::bilateralGraph(...)` 对应 GPU 端的 `Bilateral.glsl`。

它的权重形式非常典型：

1. 空间权重 `wSpatial`
2. 数值权重 `wRange`

总权重：

$$
w = w_{spatial} \cdot w_{range}
$$

其中：

$$
w_{spatial} = \exp\left(-\frac{\|x_j-x_i\|^2}{2\sigma_s^2}\right)
$$

$$
w_{range} = \exp\left(-\frac{(v_j-v_i)^2}{2\sigma_r^2}\right)
$$

这个设计的意义是：

1. 几何上近的邻居更重要
2. 数值差异太大的邻居会被抑制

因此它既有平滑能力，又有边缘保持能力。

### 14.7 第四阶段：多尺度分解

系统不是只做一次滤波，而是分层做。

设：

1. `smooth[0] = source`
2. `smooth[1]` 是第一层平滑
3. `smooth[2]` 是第二层平滑
4. `smooth[3]` 是第三层平滑

则细节层定义为：

$$
detail_l = smooth[l] - smooth[l+1]
$$

也就是说：

1. 低层细节更偏高频
2. 高层细节更偏中尺度变化

### 14.8 第五阶段：多尺度融合

融合阶段由 `MultiScaleFuse.glsl` 完成。

它的思路不是把所有细节直接加回去，而是：

1. 先估计细节强度
2. 再根据 `edgeSigma` 做抑制
3. 再乘以各层增益 `detailGain`
4. 最后加回 base

输出形式可以理解为：

$$
fused = base + w_0 d_0 + w_1 d_1 + w_2 d_2
$$

其中权重不是固定常数，而是跟局部细节幅值相关。

这样做的好处是：

1. 大特征区域保留更多细节
2. 弱噪声区域不过度增强

### 14.9 输出数组命名

这一模块会生成一批数组：

1. 平滑层：`源数组_ms_s1_P/C`、`_ms_s2_`、`_ms_s3_`
2. 细节层：`源数组_ms_d0_P/C`、`_ms_d1_`、`_ms_d2_`
3. base：`源数组_ms_base_P/C`
4. 融合结果：`源数组_ms_fused_P/C`

这对老师汇报时很有帮助，因为你可以明确说：

> 系统不是只生成一个“优化后结果”，而是把多尺度分解的中间结果也保留下来，方便分析每一层的作用。

### 14.10 这个模块目前应该怎么解释

当前最稳妥的说法不是“提出了全新的多尺度理论”，而是：

> 本模块基于图双边滤波与多尺度分解思想做工程实现，目标是提升 CAE 场数据的平滑性与可视化适配性，同时尽量保留局部特征。

这句话比较符合现在代码的真实状态。

## 15. 数据优化模块的文献基础该怎么说

你之前担心数据优化模块“缺少文献支撑”，这个担心是合理的，但并不是无解。

更稳妥的文献口径是：

1. 双边滤波基础来自 Tomasi 和 Manduchi。
2. 两尺度分解、基底层和细节层的思想可以联系 Durand 和 Dorsey。
3. 多尺度金字塔与融合思想可以联系 Burt 和 Adelson、Burt 和 Kolczynski。

也就是说，当前模块的理论基础不是“某一篇完全一一对应的论文”，而是：

> 双边滤波 + 多尺度分解 + 细节加权融合 这三类成熟思想的工程组合。

所以你汇报时完全可以说：

> 数据优化模块不是从零发明一种全新滤波器，而是在经典边缘保持滤波与多尺度融合思想基础上，结合本系统图邻域结构和 GPU 实现做出的工程化设计。

## 17. 测试程序为什么重要

这也是你论文结构里最应该保留的设计。

appendAnalyticScalarBenchmarks(...)

它回答的是：

## 22. 老师可能会追问的问题，以及建议回答

### 问题 2：为什么还要和 VTK 比

建议回答：

> VTK 是成熟工程参考实现，我用它做工程对照。但我把解析场 exact gradient 作为主真值验证，因为 VTK 并不是严格真值。

### 问题 3：你的方法和 VTK 是不是一样

建议回答：

> 不是。VTK 更偏单元导数和邻接单元平均的路线，我当前系统的非结构网格主方法是局部邻域上的自适应加权最小二乘重构。

### 问题 4：数据优化模块是不是有充分文献支撑

建议回答：

> 它不是凭空设计的，理论基础主要来自双边滤波、边缘保持平滑和多尺度分解融合这些成熟思想。当前模块更偏工程实现，后续还可以进一步补充更严格的实验和文献对齐。

### 问题 5：`notch_stress` 为什么效果不好

建议回答：

> 这个样例是高纵横比薄体上的真实应力场，既有几何困难，也有真实场不平滑的问题。它更像是在暴露当前 AWLS 对困难真实样例的局限，而不是说明整个梯度主链路都失效。解析场验证说明主框架本身是成立的。

### 问题 6：当前版本够不够本科毕设

建议回答：

> 从系统完整性、核心模块实现和实验验证角度看是够的，但在论文结论里必须诚实说明与 VTK 相比仍存在困难样例偏差。

## 25. 最后的总结

如果把整个项目压缩成一段汇报用的话，可以这样说：

> 本项目围绕 CAE 后处理中的数据预处理问题，构建了一个基于 OpenGL 的原型系统。系统以 VTK 数据为输入，通过 `VTKDataConverter` 转换为统一的内部数据结构，再由 `CAEProcessingFacade` 调度核心算法。梯度计算方面，规则网格采用有限差分，非结构网格采用自适应加权最小二乘，通过局部邻域构造、谱分析、局部维数识别和质量驱动正则增强来提高稳定性；数据优化方面，采用图双边滤波与多尺度融合，对原始场进行边缘保持平滑和细节重构。系统已经具备数据加载、计算、显示、导出和实验评估的完整流程。当前解析场实验表明主算法路线是成立的，而复杂真实样例仍暴露出一定局限，这些问题将作为后续优化方向继续完善。
