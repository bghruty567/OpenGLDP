# OpenGLDP 系统使用说明

## 1. 文档目的

本文档用于说明当前 `OpenGLDP` 项目的：

- 运行环境与依赖
- 编译与启动方式
- GUI 使用流程
- 梯度计算模块的实际使用方式
- 多尺度优化模块的使用方式
- 测试程序的典型命令
- 常见问题与排查建议

本说明文档已同步到当前实现，尤其更新了非结构网格梯度模块的实际执行逻辑：

- `Auto` 对非结构网格默认走 `AdaptiveWeightedLeastSquares`
- 但内部会优先尝试“局部最小二乘算子缓存”
- 只有缓存不可用时，才退回大邻域 WLS / Adaptive WLS

---

## 2. 系统概述

`OpenGLDP` 是一个面向 CAE 数据预处理与可视化的桌面系统，当前主要支持：

- 读取 VTK legacy 数据文件（`.vtk`）
- 将 VTK 数据转换为内部统一数据结构 `DataObject`
- 对规则网格和非结构网格计算梯度
- 对点数据或单元数据进行图双边滤波
- 执行基于双边滤波的多尺度分解与融合
- 在 Qt + VTK 界面中查看数据与结果
- 将结果重新导出为 `.vtk`

当前工程包含 3 个主要可执行目标：

- `opengldp_gui`
- `opengldp_benchmark`
- `opengldp_multiscale_test`

## 5. 编译与构建

### 5.1 推荐构建方式

在项目根目录执行：

```powershell
cmake -S . -B build `
  -DVTK_DIR="C:\你的VTK路径\lib\cmake\vtk-9.5" `
  -DGLAD_ROOT="C:\你的glad路径"
```

然后编译：

```powershell
cmake --build build --config Release
```

### 5.2 编译后目标

通常可得到：

- `opengldp_gui.exe`
- `opengldp_benchmark.exe`
- `opengldp_multiscale_test.exe`

### 5.3 关于 `Shaders` 与 `Data`

工程里已经有 `POST_BUILD` 复制逻辑，会把：

- `Shaders/`
- `Data/`

复制到目标程序目录。

最稳妥的运行方式是：

1. 直接在生成目录中运行程序，或
2. 把工作目录设置为项目根目录

否则程序可能找不到 `Shaders` 或 `Data`。

## 6. GUI 使用说明

## 6.1 启动程序

运行 `opengldp_gui.exe` 后，系统会依次初始化：

- OpenGL 上下文
- 计算着色器
- Qt 主界面
- VTK 渲染窗口

若初始化失败，一般会：

- 弹出提示框
- 在日志区打印失败原因

## 6.2 界面区域

主窗口大致分为三部分：

1. 左侧面板  
   文件加载、数据集列表、摘要信息。

2. 中间参数区  
   梯度参数和多尺度参数。

3. 右侧显示区  
   上方为 VTK 渲染窗口，下方为日志输出。

## 6.3 数据加载

点击 `Open VTK` 后：

1. 选择 `.vtk` 文件
2. 系统调用 `CAEProcessingFacade::loadDatasetFromVTKFile(...)`
3. VTK 数据被转换为 `DataObject`
4. 左侧数据集列表新增记录
5. 字段列表自动刷新

补充说明：

- GUI 默认只加载原始 VTK 数组与后续计算结果，不再自动追加 `benchmark_linear / benchmark_quadratic / benchmark_trig` 这类解析 benchmark 数组。
- 如果需要解析 benchmark，只能在测试程序 `opengldp_benchmark` 中显式打开 `--analytic-bench` 开关。

建议优先用 `Data/` 中样例测试：

- `Data\1_0.vtk`
- `Data\hexa.vtk`
- `Data\SampleStructGrid.vtk`
- `Data\ShipHull_0.vtk`

## 7. 梯度计算模块使用说明

这是当前项目最需要特别说明的部分。

## 7.1 基本概念

系统会根据：

- 网格类型
- 字段关联方式
- 方法选择

自动决定梯度的执行路径。

### 7.1.1 规则网格

规则网格使用：

- `FiniteDifference`

### 7.1.2 非结构网格

非结构网格对外仍属于最小二乘法体系，但内部会先按数据关联和单元维度分流：

1. `3D 体网格 + 点数据`
   - 优先走 `computeVolumePointGradientByCellPatches(...)`
   - 即“单元局部导数 + 点 patch 恢复”路线，更接近 `vtkGradientFilter`
2. `2D 表面网格 + 点数据`
   - 优先走局部最小二乘算子缓存
   - 即 `buildPointGradientOperator(...) + applySparseGradientOperator(...)`
3. `单元数据`
   - 优先走“单元到点稀疏重构 + 单元局部梯度算子”
   - 即 `buildPointValueReconstructionFromCells(...) + reconstructSparseValues(...) + buildCellGradientOperator(...) + applySparseGradientOperator(...)`
4. `1D 线网格 / 退化情况 / 缓存构造失败`
   - 再退回 `computeUnstructuredWLS(...)` 或 `computeUnstructuredAdaptiveWLS(...)`

这意味着：

- 你在界面里看到的 `WeightedLeastSquares / AdaptiveWeightedLeastSquares`
- 与实际内部执行路径不完全一一对应

当前实现中，`Auto` 在非结构网格上的默认方法是：

- `AdaptiveWeightedLeastSquares`

但这并不表示一定直接跑大邻域 Adaptive WLS。

### 7.1.3 解析 benchmark 开关

- `CAEProcessingFacade` 现在默认关闭解析 benchmark 注入。
- GUI 不调用这个开关，因此界面里不会出现 `benchmark_*` 和 `*_exact_grad` 这组测试字段。
- `opengldp_benchmark` 需要显式传入 `--analytic-bench` 才会在加载后追加这些数组。

## 7.2 参数区说明

梯度计算模块主要涉及以下控件：

| 控件 | 作用 |
| --- | --- |
| `Association` | 选择 `Point` 或 `Cell` |
| `Array` | 选择输入数组 |
| `Method` | 选择 `Auto / FiniteDifference / WeightedLeastSquares / AdaptiveWeightedLeastSquares` |
| `WLS Exponent` | 距离权重指数 |
| `WLS Lambda` | 正则项 |
| `Visible Component` | 控制显示分量 |

说明：

- GUI 当前显示的梯度方法选项以主流程为主。
- 更细的自适应参数，例如 `minNeighbors`、`targetNeighbors`、`planeEigenRatio` 等，当前主要通过代码或测试程序里的请求结构控制。

## 7.3 推荐使用步骤

1. 先加载数据文件
2. 选择 `Association`
3. 选择输入数组
4. 选择方法
5. 点击 `Compute Gradient`
6. 查看日志中的结果数组名、CPU 时间、GPU 时间
7. 在渲染窗口中切换结果数组观察

## 7.4 点数据与单元数据的区别

### 点数据

点数据梯度更接近：

- 单元局部导数
- 再对 incident-cell 导数做平均

### 单元数据

单元数据梯度当前优先路线是：

1. 先把单元值稀疏重构到点上
2. 再对每个单元中心应用局部梯度算子

因此：

- 单元数据路径比点数据多一步“重构”
- 对强跳变单元场，它通常比点数据更容易出现与 `vtkGradientFilter` 的差异

## 7.5 结果命名

梯度结果命名格式为：

```text
<源数组名>_grad_<P或C>_<FD或WLS或AWLS>
```

例如：

- `pressure_grad_P_FD`
- `temperature_grad_C_WLS`
- `S_Mises_grad_C_AWLS`

需要注意的是：

- 结果名中的 `AWLS` 表示“对外方法选择”
- 真实内部路径仍可能是局部最小二乘算子缓存

## 7.6 关于误差的正确理解

如果你用 `TestGradient.cpp` 对比 `vtkGradientFilter`，不要只看相对误差。

需要同时看：

- `NRMSE_vec`
- `SoftRel_P90`
- `Angle_P90_deg`
- `ScaleBias`

建议补充看：

- `NMAE_vec`
- `LowRefCount`

原因是：

- 当前输出已经改成更稳健的向量指标组合，不再把旧的逐分量相对误差作为主判断依据。
- 当 `LowRefCount` 很高时，说明低参考量级区域很多，此时 `SoftRel_*` 统计的解释需要更谨慎。

对于 `ShipHull_0.vtk` 这类：

- mixed-cell
- 含孤立点
- 单元标量跳变强

的数据集，更应当先判断误差是不是来自算法原理差异，而不是只看一个相对误差数字。

## 8. 多尺度优化模块使用说明

## 8.1 功能概述

多尺度优化模块实现的是：

- 图双边滤波
- 多尺度分解
- 细节层加权融合

## 8.2 参数说明

| 控件 | 作用 |
| --- | --- |
| `MS Levels` | 分解层数，当前建议 1~3 |
| `MS Iter/Level` | 每层双边滤波迭代次数 |
| `Spatial Sigma Factor` | 空间尺度因子 |
| `Range Sigma Factor` | 值域尺度因子 |
| `Level Scale` | 每层空间尺度放大倍数 |
| `Edge Sigma Factor` | 融合时边缘抑制参数 |
| `Detail Gain L0/L1/L2` | 各层细节增益 |
| `Store intermediate smooth/detail arrays` | 是否保存中间数组 |

## 8.3 使用步骤

1. 加载数据集
2. 选择 `Association` 和 `Array`
3. 设置多尺度参数
4. 点击 `Run MultiScale Optimization`
5. 查看日志和结果数组
6. 如需要，勾选中间结果写回

## 8.4 输出结果命名

平滑层：

```text
<源数组名>_ms_s<层号>_<P或C>
```

细节层：

```text
<源数组名>_ms_d<层号>_<P或C>
```

基底层：

```text
<源数组名>_ms_base_<P或C>
```

融合结果：

```text
<源数组名>_ms_fused_<P或C>
```

### 命令格式

### 示例


### 典型输出

- `Dataset=...`
- `Available arrays (...)`
- `VTK_time_ms_avg`
- `GL_wall_ms_avg`
- `GL_gpu_ms_avg`
- `VecErr_MAE_abs`
- `VecErr_RMSE_abs`
- `VecErr_MAX_abs`
- `RefScale_RMS`
- `NMAE_vec`
- `NRMSE_vec`
- `SoftRel_tau`
- `SoftRel_mean`
- `SoftRel_median`
- `SoftRel_P90`
- `LowRefCount`
- `Angle_mean_deg`
- `Angle_P90_deg`
- `ScaleBias`

### 10.1.1 误差指标含义与建议范围

- `VecErr_MAE_abs / VecErr_RMSE_abs / VecErr_MAX_abs`：向量绝对误差，直接比较 `||g_gl - g_ref||` 的大小，保留原始物理量纲。它们适合同一数据集、同一 benchmark 下横向比较不同算法或不同参数，但不适合跨不同量纲的数据集设统一阈值。
- `RefScale_RMS`：参考梯度场的均方根尺度，用作归一化基准。本身不是“好坏”指标，而是帮助判断当前数据集梯度量级有多大。
- `NMAE_vec`：全局归一化平均绝对误差，定义为 `sum ||g_gl-g_ref|| / sum ||g_ref||`。建议区间是：`< 0.1` 很好，`0.1-0.3` 较好，`0.3-0.5` 可接受，`0.5-1` 对当前项目阶段仍可用，`> 1` 说明平均误差已经接近或超过参考梯度量级。
- `NRMSE_vec`：全局归一化均方根误差，定义为 `sqrt(sum ||g_gl-g_ref||^2 / sum ||g_ref||^2)`。它比平均误差更敏感于局部大偏差。建议区间与 `NMAE_vec` 类似，但通常会略大一些；如果持续超过 `1`，一般说明结果还不够稳定。
- `SoftRel_tau`：软相对误差使用的稳定阈值，当前取参考梯度 RMS 的 `5%`。它不是质量指标，而是为了避免参考值接近零时相对误差被无限放大。
- `SoftRel_mean / SoftRel_median / SoftRel_P90`：软相对误差统计，其中 `SoftRel_i = ||g_gl-g_ref|| / max(||g_ref||, tau)`。`median` 反映典型局部误差，`P90` 反映最差 `10%` 区域的稳定性。经验上 `median < 0.2` 较好，`P90 < 0.5` 较好，`0.5-1` 仍可接受，`> 1` 说明有较多局部区域误差偏大。
- `LowRefCount`：参考梯度模长低于 `tau` 的向量个数。这个值越大，说明数据中“低参考量级”区域越多，局部相对误差统计越需要谨慎解释。
- `Angle_mean_deg / Angle_P90_deg`：梯度方向夹角误差，单位为度。`mean` 看整体方向一致性，`P90` 看尾部风险。经验上 `< 5°` 很好，`5°-15°` 较好，`15°-30°` 可接受，`> 30°` 往往需要重点排查邻域、投影或 fallback 路径。
- `ScaleBias`：全局幅值偏置，定义为 `sum(g_gl · g_ref) / sum(||g_ref||^2)`。它接近 `1` 说明整体尺度匹配较好。经验上 `0.9-1.1` 较好，`0.8-1.2` 可接受，`< 0.8` 常表示整体偏小，`> 1.2` 常表示整体偏大。

## 12. 常见问题与排查

## 12.1 程序启动时报 OpenGL 或初始化失败

优先检查：

- 显卡驱动是否支持 `OpenGL 4.3+`
- 当前目录下是否存在 `Shaders`
- 着色器文件是否完整

## 12.2 打开文件后看不到想要的数组

先确认：

- 当前 `Association` 是否选对
- 该数组到底在 `PointData` 还是 `CellData`

## 12.3 点击梯度计算后没有结果

依次检查：

1. 是否加载了数据集
2. 是否选中了输入数组
3. 输入数组是否为空
4. 规则网格是否被强制选成不合适的方法
5. 非结构网格的邻接和单元连接是否完整

## 12.4 相对误差看起来很大，但结果图像并不离谱

这通常不是第一时间就说明算法错了。

需要先判断：

- 参考值是否很小
- 是否是强跳变单元场
- 是否是 mixed-cell 数据集
- 是否仍有 fallback WLS 区域在起作用

## 12.5 `ShipHull_0.vtk` 这类数据为什么更难

因为这类数据同时具备：

- mixed-cell
- 孤立点
- 单元场强变化

它更适合拿来检验鲁棒性，而不是作为“所有误差都必须很小”的唯一标准用例。

## 12.6 找不到 `Shaders` 或 `Data`

解决方式：

1. 在生成目录中运行程序
2. 或把工作目录改到项目根目录
3. 或检查 `POST_BUILD` 复制是否成功

---

## 13. 使用建议总结

1. 规则网格优先看 `FD`
2. 非结构网格优先用 `Auto`
3. 对点数据和单元数据分开评估
4. 不要只看相对误差
5. 对 `ShipHull_0.vtk` 这类复杂数据，先确认实际执行路径，再谈误差大小

这样更能准确判断当前结果是“仍有实现问题”，还是“主要由与 VTK 原理不同带来的剩余差异”。
