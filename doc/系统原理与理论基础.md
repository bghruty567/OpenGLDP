# OpenGLDP 系统原理与理论基础

## 1. 文档定位

本文档说明当前 `OpenGLDP` 项目中梯度计算模块与测试模块的理论基础、系统结构和验证思路。本文档对应当前代码状态，重点覆盖：

- 规则网格与非结构网格的统一数据表示；
- 点数据与单元数据的统一梯度计算路径；
- 基于 OpenGL 的加权最小二乘梯度重构；
- `TestGradient` 的验证逻辑与评价指标；
- 论文和答辩中可直接引用的文献支撑。

## 2. 系统总体结构

当前工程可分为四层：

1. 数据输入层  
   由 VTK 负责读取 `.vtk` 数据集。
2. 内部数据层  
   由 `VTKDataConverter` 将外部 `vtkDataSet` 转成内部统一结构 `DataObject`。
3. 核心算法层  
   由 `GLGradientEngine` 和 `GLFilterEngine` 负责 GPU 端梯度、滤波和多尺度计算。
4. 门面与应用层  
   由 `CAEProcessingFacade` 对外提供统一接口，GUI 与测试程序共用同一条底层路径。

这种结构的意义在于：界面、测试、导出都复用同一套算法实现，避免 GUI 和命令行各走一套逻辑。

## 3. 统一数据模型

### 3.1 `DataObject` 的角色

`DataObject` 是项目内部的统一数据容器，负责同时保存：

- 点坐标；
- 单元中心；
- 点邻接图；
- 单元邻接图；
- 点数据数组；
- 单元数据数组。

对非结构网格，邻域采用 CSR 风格的压缩存储：

```text
neighbors[offsets[i] ... offsets[i+1])
```

这种表示方式适合 GPU 顺序读取，也适合同时支持点数据和单元数据。

### 3.2 点数据与单元数据的统一抽象

本项目中，点梯度与单元梯度不再拆成两套独立算法，而是统一抽象为：

- 一组样本位置；
- 一张样本邻接图；
- 一个输入数组；
- 一个局部最小二乘重构问题。

因此：

- 点数据梯度使用“点坐标 + 点邻接”；
- 单元数据梯度使用“单元中心 + 单元邻接”。

这也是当前项目能够同时支持点数据和单元数据梯度的关键。

## 4. 梯度计算原理

## 4.1 规则网格：有限差分

对于规则网格，当前路径仍然是有限差分：

```text
computeGradient(...)
-> computeByFD(...)
-> GLGradientEngine::computeRegularFD(...)
-> Shaders/FD.glsl
```

这一路径适用于规则采样、拓扑规整的数据集，优点是简单、稳定、成本低。

## 4.2 非结构网格：加权最小二乘重构

对非结构网格，当前主路径为统一的加权最小二乘重构：

```text
computeGradient(...)
-> computeByAdaptiveWLS(...)
-> ensureAdaptiveSupport(...)
-> GLGradientEngine::computeUnstructuredAdaptiveWLS(...)
-> Shaders/AdaptiveWLS.glsl
```

其基本思想是在每个样本点或单元中心附近建立局部一阶近似：

```math
\phi(\mathbf{x}) \approx a_0 + \mathbf{g}^{\mathsf T}(\mathbf{x} - \mathbf{x}_0)
```

其中：

- `a0` 为局部常数项；
- `g` 为待求梯度。

然后最小化局部残差：

```math
\min_{a_0,\mathbf{g}}
\sum_k w_k
\left(
\phi_k - a_0 - \mathbf{g}^{\mathsf T}(\mathbf{x}_k-\mathbf{x}_0)
\right)^2
 + \lambda \|\mathbf{g}\|_2^2
```

这里：

- `wk` 为距离权重；
- `lambda` 为正则项；
- 权重与邻域形状共同决定重构精度和稳定性。

Mavriplis 与 Thomas 对非结构网格最小二乘梯度重构做了系统分析，指出线性精确性、权重设计和网格质量对精度有直接影响 [1]。Wang 等人进一步给出分析与数值证据，说明最小二乘类方法在测试的非结构网格上至少具有一阶精度，并可用解析函数、网格收敛与 MMS 验证 [2]。

## 4.3 自适应支撑构造

在直接做最小二乘之前，项目会先构造更稳健的局部支撑：

1. 从点邻接或单元邻接得到基础 stencil；
2. 邻域不足时扩展二环邻域；
3. 必要时用 KNN 补邻域；
4. 计算局部协方差矩阵；
5. 估计局部主方向、维数标签与邻域质量；
6. 对退化邻域增大正则化。

这一部分是项目的工程增强项，不是照搬某个现成库，而是在最小二乘主框架上做的稳定化设计。

## 4.4 点数据与单元数据如何统一支持

OpenFOAM 的 least-squares 梯度方案是典型的单元中心有限体积做法，文档明确说明其“cell gradient is calculated using least squares”，并提供基于不同 cell stencil 的方案 [3]。本项目借鉴的是“局部 stencil + 最小二乘重构”这条思路，但实现上做了两点扩展：

1. 不只支持单元数据，也支持点数据；
2. 统一通过 `CAEProcessingFacade` 调度，不把点路径和单元路径拆开维护。

因此，当前实现不是“把 OpenFOAM 代码直接搬过来”，而是参考同类最小二乘思想，在本项目的数据结构和 OpenGL 执行框架下重新实现。

## 4.5 向量场梯度

如果输入是三分量向量场，则输出梯度是 9 分量，按如下顺序展开：

```text
du/dx, du/dy, du/dz,
dv/dx, dv/dy, dv/dz,
dw/dx, dw/dy, dw/dz
```

这与 `vtkGradientFilter` 文档中的输出约定一致 [4]，因此在测试程序中与 VTK 对照时更直观。

## 5. OpenGL 实现思路

当前项目中，OpenGL 负责大规模局部重构的并行执行：

- CPU 端准备样本位置、偏移数组、邻居列表和局部支撑属性；
- GPU 端按样本逐个求解局部最小二乘问题；
- 结果回写到 `DataObject` 中，形成新的梯度数组。

这样设计的优点是：

- 对大规模样本计算更友好；
- 保持 CPU 端门面接口稳定；
- 便于后续继续扩展滤波与多尺度流程。

## 6. `vtkGradientFilter` 在本项目中的角色

`vtkGradientFilter` 是本项目的重要参考基线，但不是“真值”。VTK 官方文档将其描述为一个“general filter for gradient estimation”，并明确指出：

- 梯度计算依赖输入数据集类型；
- 边界上不是中心差分；
- 输出数组分量数为输入分量数的 3 倍 [4]。

因此，在本项目中应这样理解 `vtkGradientFilter`：

- 它适合作为工程参考实现；
- 它不等于解析真值；
- 当真实数据缺少解析梯度时，可以作为对照基线；
- 当存在 `*_exact_grad` 时，解析梯度优先级高于 VTK 基线。

## 7. 测试与验证方法

## 7.1 为什么要优先做解析场验证

如果数据集中存在解析函数和精确梯度，那么算法验证最直接、最干净。Wang 等人的文章就是通过解析函数、误差范数和网格收敛来分析梯度重构精度 [2]。这也是当前 `TestGradient` 新版本的核心思路：

- 先验证算法本身是否能重构已知梯度；
- 再看真实数据相对 VTK 的工程表现。

当前项目已在 `CAEProcessingFacade` 中为点/单元两类数据注入多组解析 benchmark，包括：

- 标量：`benchmark_linear`、`benchmark_quadratic`、`benchmark_trig`、`benchmark_cubic`、`benchmark_gaussian`
- 向量：`benchmark_vec_linear`、`benchmark_vec_poly`、`benchmark_vec_trig`

并为每个 benchmark 提供对应的 `*_exact_grad`。

## 7.2 推荐的验证顺序

建议按以下顺序测试：

1. 线性场精确性检查  
   看是否至少满足线性精确重构。
2. 平滑解析场测试  
   使用 quadratic、trig、cubic、gaussian 等场，检查一般精度。
3. 向量场梯度测试  
   检查 9 分量输出路径是否正确。
4. 网格加密或多数据集对照  
   观察误差是否有稳定下降趋势。
5. 真实数据与 VTK 对照  
   作为工程基线比较，而不是数学真值验证。

## 7.3 指标体系

`TestGradient` 当前使用的核心指标包括：

- `VecErr_MAE_abs`
- `VecErr_RMSE_abs`
- `VecErr_MAX_abs`
- `NMAE_vec`
- `NRMSE_vec`
- `SoftRel_mean`
- `SoftRel_P90`
- `Angle_mean_deg`
- `Angle_P90_deg`
- `ScaleBias`

这些指标覆盖了：

- 绝对误差；
- 归一化误差；
- 低参考值区域的稳健相对误差；
- 方向误差；
- 整体幅值偏差。

## 7.4 为什么真实数据与 VTK 的结果可能“不好看”

真实数据与 VTK 差距较大，并不自动说明当前模块失效，常见原因包括：

1. 真实数据没有解析真值，VTK 本身只是参考算法；
2. 非结构网格局部质量较差，最小二乘对 stencil 形状敏感 [1]；
3. 不同算法在边界处理、局部支撑、平滑效应上本来就会不同 [4]；
4. 某些真实物理场本身不够光滑，局部梯度差异会被放大。

因此，论文或报告中不应简单写成“与 VTK 不一致，所以算法不成熟”；更合理的说法是：

- 解析 benchmark 结果证明算法主路径是正确的；
- 真实复杂数据相对 VTK 仍有偏差，这属于工程对照与后续优化方向。

## 7.5 论文中建议保留的实验

参考文献中的常见做法与当前项目目标，建议至少保留四类实验：

1. 解析场精确梯度实验；
2. 网格尺度变化或不同网格集实验；
3. 复杂网格/复杂真实数据对照实验；
4. CPU/GPU 时间对比实验。

Oliver 等人在非结构有限元验证中采用了解析解和制造解相结合的方法来正式测量误差与收敛率 [5]。这也说明“解析场 + 收敛 + 工程样例”的测试组合是成熟做法，而不是临时拼凑。
