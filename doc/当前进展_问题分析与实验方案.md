# 当前进展、问题分析与实验方案

更新时间：`2026-04-18`

## 1. 文档用途

这份文档不是普通的使用说明，而是面向当前毕业设计阶段的“汇报与实验准备稿”。

它主要解决 3 个问题：

1. 帮我把当前系统已经做到什么、主线算法是什么说清楚。
2. 帮我把“为什么结果不好”从单一解释，拆成更真实的几个原因。
3. 帮我设计一套可以继续分析误差来源、并能向老师展示的实验方案。

---

## 2. 当前阶段结论

截至目前，可以先明确 4 个判断：

1. 当前系统主线已经基本收敛为：
   - 规则网格：`FD`
   - 非结构网格：`AWLS`
2. 梯度结果不理想的原因，当前判断**不只是“没有适配 2D 网格”**。
3. 对一些壳体/曲面型数据，直接用三维环境梯度做真值或对照，可能会把本来属于法向口径差异的问题也算进误差里。
4. 真实工程字段与 `vtkGradientFilter` 的比较，只能说明“与参考实现差多少”，不能直接说明“谁更接近数学真值”。

所以当前最重要的工作，不是简单继续堆算法，而是先把**误差口径、几何类型、邻域质量、参数敏感性**这几件事拆开分析清楚。

## 4. 当前算法主线与原理

### 4.1 统一数据结构 `DataObject`

之所以引入 `DataObject`，是因为直接在 VTK 原始对象上做 GPU 计算不够方便。  
当前内部统一采用扁平数组组织数据，便于：

1. 上传到 SSBO；
2. 统一管理点字段和单元字段；
3. 用同一套接口支持 GUI、测试程序和导出模块。

`DataObject` 中最关键的内容包括：

1. `points`
   - 点坐标；
2. `cellCenters`
   - 单元中心坐标；
3. `dataArrays`
   - 点/单元字段；
4. `pointNeighbors + pointNeighborOffsets`
   - 点邻域图；
5. `cellNeighbors + cellNeighborsOffsets`
   - 单元邻域图；
6. `cells + cellOffsets + cellTypes`
   - 单元拓扑连接关系。

---

### 4.2 梯度计算模块

### 4.2.1 规则网格：有限差分 FD

对规则网格，当前采用有限差分。  
它的优点是：

1. 结构清楚；
2. 邻域关系天然规则；
3. 计算效率高；
4. 对规则采样数据有较稳定的精度。

适用对象主要是：

1. `vtkImageData`
2. 其他可稳定映射到规则索引结构的数据

### 4.2.2 非结构网格：自适应加权最小二乘 AWLS

对非结构网格，当前主线是 AWLS。

基本思路是：

1. 以点或单元中心为采样位置；
2. 收集局部邻域；
3. 建立局部加权最小二乘拟合；
4. 通过拟合系数重建梯度；
5. 针对局部近共面、近共线或邻域质量差的情形，做维度自适应和正则增强。

当前 AWLS 里已经引入了几类辅助信息：

1. 自适应邻域；
2. 局部主方向框架；
3. 局部维度标签；
4. 邻域质量估计；
5. 局部平均邻距。

这样做的目的，是为了缓解下面这些常见问题：

1. 非结构网格邻域分布不均匀；
2. 壳体/曲面数据局部接近二维；
3. 边界附近邻域退化；
4. 普通三维 WLS 在病态局部系统上不稳定。

### 4.2.3 当前实现上的实际调度逻辑

在 `CAEProcessingFacade::computeGradient(...)` 中，当前对外主线已经明确为：

1. 规则网格 -> `FiniteDifference`
2. 非结构网格 -> `AdaptiveWeightedLeastSquares`

这意味着现在给老师汇报时，不建议再把项目说成“还在同时并列比较很多梯度算法”，而应该说：

> 当前系统已经收敛到一条较稳定的工程主线，后续重点是分析这条主线在哪些几何和字段上表现好，在哪些情形下仍存在误差与局限。

---

### 4.3 数据优化模块

数据优化模块当前采用“图双边滤波 + 多尺度分解与融合”的思路。

主流程是：

```text
原始字段
-> 多层图双边滤波
-> 相邻平滑层做差得到 detail
-> 用 base + detail 重建 fused
```

它的核心目标不是简单把数据“抹平”，而是尽量同时做到：

1. 抑制噪声；
2. 保留主要结构；
3. 控制高频细节的重建强度；
4. 支持定量评估和 VTK 导出可视化。

---

3. 几何诊断；

它现在特别重要的改进点在于：

3. 引入低梯度区稳健相对误差；
4. 输出局部维度统计和几何摘要。

### 4.4.2 `TestMultiScale.cpp`

现在的数据优化测试程序也不只是“跑一遍滤波”，而是形成了完整实验闭环：

1. 合成干净场；
2. 自动加噪；
3. 执行多尺度优化；

这对于论文实验和老师汇报都很重要，因为它能把：

1. 视觉上更平滑；
2. 数值上更接近真值；

这两件事区分开。

## 5. 当前遇到的问题与最新判断

### 5.1 现象

目前看到的主要现象是：

1. 一些非结构网格数据上的梯度结果偏差较大；
2. 尤其是壳体、曲面型、工程真实字段，和预期差距较明显；
3. 某些数据用 `vtkGradientFilter` 对照时，差别不小；
4. 仅仅说“没有适配 2D 网格”已经不足以解释全部问题。

---

### 5.2 当前新的判断

当前更合理的判断应该是：

> 结果不好的原因，既有算法本身在复杂局部几何上的局限，也有测试口径、参考口径和数据特性带来的误差放大问题。

也就是说，当前问题至少包含两层：

1. 算法问题；
2. 评价问题。

---

### 5.3 可能的主要原因

### 5.3.1 曲面/壳体数据上的“环境梯度”与“切空间梯度”口径不一致

对一些壳体或曲面型数据，局部采样本质上更接近二维流形。  
这时如果直接拿三维解析环境梯度作为参考，可能会出现一个问题：

1. 算法实际更擅长恢复切向变化；
2. 参考值却把法向分量也一起计入误差；
3. 于是会把“口径不一致”误认为“算法完全错误”。

这也是为什么现在测试程序里需要额外引入 intrinsic 指标。

### 5.3.2 真实工程字段缺少严格数学真值

对 `ShipHull_0`、`notch_stress` 这类真实工程字段来说，通常并没有解析真值。  
所以当前做的很多比较，本质上是在比较：

1. 我的实现；
2. `vtkGradientFilter`；

之间的差异，而不是谁更接近真正物理真值。

因此：

1. `vtk` 对照很重要；
2. 但它只能当工程基线，不能当绝对真值。

### 5.3.3 邻域退化与局部几何病态

非结构网格梯度重建高度依赖局部邻域质量。  
当出现下列情况时，误差会明显放大：

1. 邻居过少；
2. 邻域分布极不均匀；
3. 局部近共面或近共线；
4. 边界附近一侧缺邻居；
5. 单元中心分布质量不稳定。

即使已经加入 AWLS，自适应方法也不能保证所有局部都足够稳定。

### 5.3.4 低梯度区域会放大相对误差

如果真值梯度本身很小，那么哪怕绝对误差不大，相对误差也可能非常夸张。  
这会带来两个风险：

1. 报表上的相对误差看起来很差；
2. 实际上算法在高梯度主结构区可能并没有那么差。

所以现在不能只看普通相对误差，必须同时看：

1. 绝对误差；
2. soft relative error；
3. 过滤低梯度区后的 stable 指标。

### 5.3.5 点/单元关联、分量组织、ParaView 显示口径可能不一致

在做对照时，还需要警惕一些“不是算法问题”的误差来源：

1. 比较的是点数据还是单元数据；
2. 比较的是单个分量梯度还是向量场雅可比；
3. ParaView 里显示的是模长还是具体分量；
4. 色条范围是不是一致；
5. 是否把不同数量级字段放在同一口径下比较。

这些问题不会改变算法结果，但会显著影响“看起来差不差”的结论。

### 6.2 建议继续做的算法层改进

### 改进 1：更明确地区分体数据与曲面数据的评价口径

建议在论文和汇报里明确区分：

1. 体数据：
   - 以三维环境梯度为主；
2. 曲面/壳体数据：
   - 同时报告环境梯度误差与切空间梯度误差。

这样可以避免把“口径不一致”误说成“算法彻底失败”。

### 改进 2：加强邻域质量诊断

建议进一步输出或统计：

1. 邻域样本数分布；
2. 局部维度标签分布；
3. 邻域质量评分分布；
4. 边界样本比例；
5. 最差案例对应的位置索引。

这样更容易把“误差大”追溯到具体局部几何问题。

### 改进 3：做参数消融，而不是只看单一参数组合

对于 AWLS，至少需要系统比较下面这些开关/参数：

1. `useAdaptiveNeighborhood`
2. `useAdaptiveDimension`
3. `useAdaptiveRegularization`
4. `minNeighbors / targetNeighbors / maxNeighbors`
5. `radiusScale`
6. `planeEigenRatio / lineEigenRatio`
7. `lambdaAmplify`

否则很难判断当前误差究竟是“算法主线有问题”，还是“参数没有打到合适区间”。

### 改进 4：把 ParaView 可视化正式纳入实验流程

建议不要只把 `.vtk` 导出当成附加功能，而应当把它作为正式实验的一部分。

原因是：

1. 某些局部错误很难仅靠整体误差指标看出来；
2. 某些整体指标不差，但局部边界失真很严重；
3. 多尺度优化是否过度平滑，也更适合在 ParaView 中观察。

## 7. 详细实验方案

下面的实验设计目标，是把“误差来源”一步步拆开，而不是把所有结果混在一起看。

### 7.1 实验 A：解析场基线精度实验

### 目的

验证在“有真值、几何较清晰”的条件下，当前梯度主线本身是否正确。

### 数据

建议优先用带 benchmark 的数据集，例如：

1. `AngularSector`
2. 其他已能稳定注入解析场的数据集

### 变量

1. 点数据 / 单元数据
2. 标量场 / 向量场
3. `benchmark_linear`
4. `benchmark_quadratic`
5. `benchmark_trig`
6. `benchmark_cubic`
7. `benchmark_gaussian`

### 指标

1. `MAE`
2. `RMSE`
3. `NMAE`
4. `NRMSE`
5. `soft relative error`
6. `angle mean / P90`

### 目的性结论

这一组实验主要回答：

1. 算法在理想解析场上是否基本正确；
2. 高阶场和振荡场会不会显著恶化；
3. 标量场和向量场差异大不大。

### 建议命令

```text
opengldp_benchmark --dataset=Data\AngularSector.vtk --assoc=cell --run=benchmarks --reference=analytic --csv=results\expA_analytic.csv
```

---

### 7.2 实验 B：曲面/壳体数据的 ambient 与 intrinsic 对比实验

### 目的

验证“结果不好是否部分来自参考口径不一致”。

### 数据

优先选择明显具有曲面/壳体特征的数据。

### 做法

对同一个 case，同时记录：

1. ambient 指标；
2. intrinsic 指标。

### 观察重点

如果出现下面现象：

1. ambient 误差很大；
2. intrinsic 误差明显改善；

那么就说明当前问题并不完全是算法失效，而是“曲面问题被三维环境梯度口径过度惩罚”。

### 目的性结论

这组实验非常适合向老师解释：

> 当前结果差，不一定只是算法没有适配 2D，而更可能是局部流形维度与评价口径没有拆开。

---

### 7.3 实验 C：AWLS 参数消融实验

### 目的

分析当前误差到底更敏感于哪一类参数。

### 变量分组

建议至少做下面 4 组消融：

1. 固定其他参数，只开关 `adaptiveNeighborhood`
2. 固定其他参数，只开关 `adaptiveDimension`
3. 固定其他参数，只开关 `adaptiveRegularization`
4. 分别扫描邻域参数和阈值参数

### 推荐扫描区间

1. `minNeighbors`：`8 / 12 / 16`
2. `targetNeighbors`：`16 / 20 / 24 / 32`
3. `maxNeighbors`：`24 / 32 / 40`
4. `radiusScale`：`1.5 / 2.0 / 2.5 / 3.0`
5. `planeEigenRatio`：`0.03 / 0.06 / 0.10`
6. `lineEigenRatio`：`0.01 / 0.02 / 0.04`
7. `lambdaAmplify`：`1 / 2 / 4 / 8`

### 重点看什么

1. 哪些参数对误差改善最明显；
2. 哪些参数会显著影响时间开销；
3. 哪些参数对曲面型数据特别敏感；
4. 是否存在明显的稳定区间。

### 目的性结论

这组实验回答的是：

1. 问题主要是“算法框架不对”；
2. 还是“同一框架下参数还没调到合适区间”。

---

### 7.4 实验 D：真实工程字段与 VTK 对照实验

### 目的

衡量当前实现相对工程参考实现的差异，并找出困难样例。

### 数据

建议优先分析：

1. `ShipHull_0`
2. `notch_stress`
3. 其他真实工程数据集中的代表字段

### 做法

1. 只把 `vtkGradientFilter` 当作参考基线；
2. 不把这组实验直接写成“真值误差”；
3. 同时输出几何摘要与误差摘要。

### 重点看什么

1. 哪些字段与 VTK 最接近；
2. 哪些字段偏差最大；
3. 偏差大的样例是否同时具有：
   - 曲面型几何；
   - 邻域退化；
   - 边界复杂；
   - 低梯度占比高。

### 建议命令

```text
opengldp_benchmark --dataset=Data\ShipHull_0.vtk --assoc=point --run=fields --reference=vtk --analytic-bench=off --csv=results\expD_ship_vs_vtk.csv
```

### 目的性结论

这一组实验适合汇报时用来说明：

1. 当前系统已经能跑真实工程数据；
2. 但与成熟参考实现相比，复杂样例上仍有差距；
3. 差距主要集中在哪类数据。

---

### 7.5 实验 E：数据优化模块 synthetic 去噪实验

### 目的

验证多尺度优化是否真的能把加噪输入拉回真值附近。

### 数据

用 `TestMultiScale` 自动生成的 clean 场。

### 噪声类型

1. `gaussian`
2. `impulse`
3. `mixed`

### 指标

1. `input_mae / fused_mae`
2. `input_rmse / fused_rmse`
3. `input_nrmse / fused_nrmse`
4. `mae_improvement_ratio`
5. `rmse_improvement_ratio`
6. `roughness_ratio`

### 结果解释

1. 若误差下降且 roughness 合理下降，说明优化有效；
2. 若 roughness 大幅下降但误差没有改善，说明可能过度平滑；
3. 若误差改善但局部结构被破坏，则需要结合 ParaView 再判断。

### 建议命令

```text
opengldp_multiscale_test --dataset=Data\ShipHull_0.vtk --assoc=point --run=synthetic --noise=all --export=results\expE_vtk --csv=results\expE_multiscale.csv
```

---

### 7.6 实验 F：真实字段多尺度可视化实验

### 目的

在没有真值时，观察多尺度优化是否保留了主结构，是否存在过度平滑。

### 数据

真实工程字段，例如：

1. 位移
2. 应力
3. 其他高噪声或高梯度字段

### 做法

1. 导出输入字段、base、smooth、detail、fused；
2. 在 ParaView 中统一色条范围；
3. 对比关键区域。

### ParaView 重点看什么

1. 高频噪声是否被抑制；
2. 主边界是否被抹平；
3. 尖峰结构是否被削弱过头；
4. 各层 detail 是否具有明确的尺度分工。


### 7.7 如何区分“参考口径问题”和“算法本身问题”

后续做实验时，建议按下面逻辑判断：

### 更像参考口径问题的表现

1. ambient 指标差，但 intrinsic 指标明显改善；
2. 真实字段与 VTK 差距大，但解析 benchmark 表现还可以；
3. 误差主要集中在法向分量或低梯度区域；
4. 不同显示口径下视觉差异很大。

### 更像算法本身问题的表现

1. 解析 benchmark 本身就明显失真；
2. 不同参考口径下都差；
3. 误差集中在邻域退化或边界区；
4. 参数稍微变化就剧烈波动；
5. 非有限值、异常大值较多。

## 10. 现阶段最重要的事情

当前最重要的，不是急着下结论说“算法一定不行”或者“一定只是 2D 适配问题”，而是把下面三件事做扎实：
