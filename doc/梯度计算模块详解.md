# 梯度计算模块详解

## 1. 当前版本的算法结构

当前梯度模块已经不是“很多网格类型各走一条独立算法线”的结构了。

现在只有两条公开主线：

- 规则网格：有限差分 `FD`
- 非结构网格：统一自适应加权最小二乘 `AWLS`

## 2. 总入口

总入口是：

```cpp
CAEProcessingFacade::computeGradient(...)
```

其逻辑可以写成：

```text
if dataset is regular:
    use FD
else:
    use AWLS
```

## 3. 规则网格梯度

规则网格执行链：

```text
computeGradient(...)
-> computeByFD(...)
-> GLGradientEngine::computeRegularFD(...)
-> Shaders/FD.glsl
```

特点：

- 适用于 `vtkImageData / vtkRectilinearGrid / vtkStructuredGrid`
- 点数据和单元数据都支持
- 样本位置直接取网格点或单元中心

## 4. 非结构网格梯度的统一流程

非结构网格执行链：

```text
computeGradient(...)
-> computeByAdaptiveWLS(...)
-> ensureAdaptiveSupport(...)
-> GLGradientEngine::computeUnstructuredAdaptiveWLS(...)
-> Shaders/AdaptiveWLS.glsl
```

失败时内部回退：

```text
GLGradientEngine::computeUnstructuredWLS(...)
-> Shaders/WLS.glsl
```

## 5. 支撑构造到底做了什么

当前 `ensureAdaptiveSupport(...)` 的核心工作是把原始邻接图整理成适合 GPU 最小二乘求解的局部支撑。

### 5.1 候选邻域

先取基础邻接：

- 点数据用点邻接
- 单元数据用单元邻接

如果开了 `useAdaptiveNeighborhood` 且邻居不够：

1. 先扩二环邻接
2. 再用 KNN 补邻域

### 5.2 局部尺度

程序会先估计：

- 局部平均邻居距离
- 全局平均邻居距离

这样做是为了在补 KNN 时使用更合理的局部搜索半径，而不是所有地方都用同一个固定尺度。

### 5.3 局部谱分析

在候选邻域上计算局部协方差矩阵并做特征分解，得到：

- 主方向框架 `frame`
- 局部维数标签 `dimTag`
- 邻域质量 `quality`
- 平均邻居距离 `meanNeighborDistance`

其中 `dimTag` 的含义是：

- `1` 更像线状支撑
- `2` 更像面状支撑
- `3` 更像体状支撑

### 5.4 邻域平衡

代码还会在局部框架下把候选点按方向分桶，尽量保证邻域不是“都堆在一侧”。

这一步的目的不是追求复杂技巧，而是减少最小二乘矩阵病态和偏置。

## 6. GPU 求解阶段做了什么

支撑构造完成后，GPU 侧读取：

- 样本坐标
- 邻域偏移
- 邻居列表
- 局部框架
- 维数标签
- 邻域质量
- 平均邻居距离

然后在 `AdaptiveWLS.glsl` 中完成带权最小二乘求解。

当前可调机制包括：

- 距离权重指数
- 正则参数
- 维数感知
- 低质量邻域时的正则增强

## 7. 点数据与单元数据为什么还能共用一条线

因为现在它们都被抽象成了同一种形式：

```text
样本位置 + 邻接关系 + 局部支撑属性 + GPU 最小二乘求解
```

区别只是样本和邻接来源不同：

- 点数据：点与点
- 单元数据：单元中心与单元中心

## 8. 为什么真实数据可能和 VTK 仍有明显差异

即使解析场表现已经不错，真实数据相对 `vtkGradientFilter` 仍可能出现更大误差，常见原因包括：

1. 真实数据没有 exact gradient，VTK 只是参考实现，不是真值。
2. `ShipHull_0`、`notch_stress` 这类数据往往几何更复杂，局部网格质量更差。
3. 非结构网格上的最小二乘方法对邻域形状、尺度拉伸和局部维数退化都比较敏感。

所以：

- 解析场验证更能说明算法本身是否正确。
- 真实数据对 VTK 的偏差更适合写成工程对照和局限分析。

## 9. 为什么这一版更适合本科毕设

这一版相比旧版本更适合定稿，主要因为：

1. 算法口径统一了。
2. 默认主线更接近“纯 OpenGL 计算”。
3. 文档和代码现在能一一对应。
4. 解析场验证、真实数据对照和性能测试三类实验都能围绕同一套主线展开。

## 10. 论文里建议用的伪代码

```text
输入: 样本位置 x_i, 邻接图 N(i), 标量/向量场 u
若为规则网格:
    直接做有限差分
否则:
    为每个样本构造局部邻域
    必要时扩二环并用 KNN 补足
    做局部协方差谱分析，得到 frame、dimTag、quality
    在 GPU 上执行带权最小二乘
    输出梯度
```

## 11. 和当前论文验证的关系

如果你现在的实验现象是：

- 解析场上所有当前测试数据 `NMAE_vec < 1`、`NRMSE_vec < 1`
- 真实数据里 `ShipHull_0` 和 `notch_stress` 相对 VTK 还在 `1` 到 `2` 之间

那么最合适的论文结论不是“完全替代 VTK”，而是：

> 当前统一 AWLS 方案已经可以作为本科毕设的最终工程方案，但在复杂真实非结构网格上，相对 VTK 仍存在困难样例偏差，这部分需要作为局限与后续工作诚实报告。
