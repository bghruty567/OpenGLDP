## 2. 项目当前已经定下来的主线

当前梯度模块的公开主线已经收敛为两条：

- 规则网格：`FD`
- 非结构网格：统一 `AWLS`

对应结论如下：

1. `Auto` 模式下：
   - 规则网格自动走 `FiniteDifference`
   - 非结构网格自动走 `AdaptiveWeightedLeastSquares`
2. 非结构网格不再对外维持多套按单元类型切换的公开算法主线。
3. 非结构网格结果命名统一按 `AWLS` 口径描述。
4. `benchmark` 中的解析场数组注入只用于测试程序，不是 GUI 正常使用流程的一部分。
5. GUI 当前公开的方法名应写为：
   - `Auto`
   - `FiniteDifference`
   - `AdaptiveWLS`

## 3. 关键代码位置

当前关键实现位置主要在：

- `CAEProcessingFacade.cpp`
  - `computeGradient(...)`
  - `computeByAdaptiveWLS(...)`
  - `buildAdaptiveGradientSupport(...)`
  - `makeResultName(...)`
- `GLGradientEngine.cpp`
- `TestGradient.cpp`
- `app/MainWindow.cpp`

已确认的行为：

1. `computeGradient(...)` 中，`Auto` 会根据网格类型自动路由到 `FD` 或 `AWLS`。
2. `computeByAdaptiveWLS(...)` 中，先构造自适应支撑，再调用 `computeUnstructuredAdaptiveWLS(...)`。
3. 若 `AdaptiveWLS` 失败，内部仍可能回退到普通 `WLS` 作为兜底。
4. `buildAdaptiveGradientSupport(...)` 的支撑构造流程大致为：
   - 一环邻接
   - 必要时二环扩邻
   - KNN 补邻域
   - 局部协方差谱分析
   - 得到 `frame / dimTag / quality / meanNeighborDistance`
   - 再做较平衡的邻居选择

## 6. 当前最重要的未解决问题

当前最突出的实际问题是：`Data/notch_stress.vtk` 上的真实场梯度结果明显异常。

最近一次用户测试的关键信号是：

- `AnalyticBenchmarks=OFF`
- `Reference=VTK gradient filter`
- `NMAE_vec=1.90725`
- `NRMSE_vec=1.83175`
- `Angle_mean_deg=68.7611`
- `Angle_P90_deg=109.725`
- `ScaleBias=0.107697`

而且从逐点输出看，存在非常明显的异常现象：

1. `GL` 结果大量出现巨大的负 `z` 分量，量级常在 `-1e9`
2. `REF` 大多只有 `1e7 ~ 1e8` 量级
3. 这不是正常的“小偏差”，而是明显跑歪

当前更合理的判断不是“程序完全坏了”，而是：

1. 解析场主管线大概率是通的
2. 真正的问题集中在真实应力场上的稳定性
3. 尤其可能出在表面/壳/薄层/高阶单元类数据上的法向分量处理

## 7. 对 `notch_stress` 异常的当前怀疑

目前优先怀疑以下几类原因：

1. `Nodal Stress-0` 本身是噪声较强、非光滑的有限元后处理标量场。
2. 数据几何可能更接近表面/壳/薄层结构，而当前通用 `AWLS` 在这类数据上产生了异常法向分量。
3. `vtkGradientFilter` 与当前点邻域最小二乘在真实复杂场上的口径不完全一致。
4. 当前支撑构造、维数判别、局部坐标框架或正则方式对该样例还不够稳。

## 8. 当前最推荐的后续诊断路线

如果后续 AI 要继续接手，建议按下面顺序推进：

1. 先加诊断输出，而不是继续只看误差指标。
2. 优先在这些位置补统计：
   - `CAEProcessingFacade.cpp` 的支撑构造相关逻辑
   - `TestGradient.cpp`
3. 建议输出的诊断信息包括：
   - `dimTag` 分布
   - `quality` 的最小/最大/均值
   - 邻居数最小/最大/均值
   - 有多少点被判成 3D
   - 是否存在大量异常法向或第三主轴方向
4. 在同一网格上分别跑两次：
   - 一次 `--analytic-bench`
   - 一次真实数组 `Nodal Stress-0`
5. 如果结论是“同网格解析场正常、真实场异常”，那就说明问题集中在真实场稳定性，而不是几何主链路整体失效。
6. 若进一步确认是表面/壳数据法向问题，可考虑：
   - 对 `dimTag == 2` 的点加入切向约束
   - 或在局部二维坐标中做拟合，只恢复面内梯度后再映回 3D

## 10. 使用偏好与协作约束

这是后续 AI 很容易忽略、但实际上非常重要的背景：

1. 用户偏好中文交流。
2. 这是本科毕设场景，论文口径必须诚实、务实，不能夸大。
3. 用户非常在意“解析 benchmark 只用于测试程序，不进入 GUI 日常流程”。
4. 用户也非常在意文献支撑要能落到：
   - 哪篇文献
