## 2. 当前已经定下来的主线

当前梯度模块的公开主线已经收敛为两条：

- 规则网格：`FiniteDifference`
- 非结构网格：统一 `AdaptiveWeightedLeastSquares`，简称 `AWLS`

对应结论：

1. `Auto` 模式下：
   - 规则网格自动走 `FiniteDifference`
   - 非结构网格自动走 `AdaptiveWeightedLeastSquares`
2. 非结构网格不再对外维护多套按单元类型切换的公开算法主线。
3. GUI 当前公开的方法名应保持为：
   - `Auto`
   - `FiniteDifference`
   - `AdaptiveWLS`
4. 解析 benchmark 只用于测试程序与论文验证，不属于 GUI 日常使用流程。
5. 论文口径必须诚实，不能写成“已经全面等价 VTK/ParaView”。

## 3. 当前关键代码位置

- `CAEProcessingFacade.cpp`
  - `computeGradient(...)`
  - `computeByAdaptiveWLS(...)`
  - `buildAdaptiveGradientSupport(...)`
  - `saveDatasetToVTKFile(...)`
- `GLGradientEngine.cpp`
- `shaders/AdaptiveWLS.glsl`
- `TestGradient.cpp`
- `TestMultiScale.cpp`
- `app/MainWindow.cpp`

当前已确认的行为：

1. `computeGradient(...)` 中，`Auto` 会根据网格类型自动路由到 `FD` 或 `AWLS`。
2. `computeByAdaptiveWLS(...)` 中，先构造自适应支撑，再调用 OpenGL 的 `computeUnstructuredAdaptiveWLS(...)`。
3. 如果 `AdaptiveWLS` 路径失败，内部仍可能回退到更普通的 `WLS` 作为兜底。
4. `buildAdaptiveGradientSupport(...)` 的大致流程为：
   - 一环邻接
   - 必要时二环扩邻
   - KNN 补邻
   - 局部协方差谱分析
   - 得到 `frame / dimTag / quality / meanNeighborDistance`
   - 再做方向更平衡的邻居筛选
5. `AdaptiveWLS.glsl` 已支持按局部谱信息做 1D/2D/3D 自适应求解，而不是无脑固定三维。

## 4. 近期已经确认并完成的代码改动

### 4.1 导出 VTK 文件问题已修复

此前导出的 `.vtk` 文件有两类问题：

1. `vtkDataSetWriter` 默认写出了偏新的 legacy 5.1 风格，部分 ParaView/旧 reader 读不稳。
2. ASCII 导出时如果字段里含 `NaN`，更容易造成外部读取异常或表现怪异。

已完成的修复：

1. 在 `CAEProcessingFacade::saveDatasetToVTKFile(...)` 中强制 legacy 版本为 `VTK 4.2`。
2. GUI 导出改为 binary。
3. `TestMultiScale.cpp` 中的导出也改为 binary。

用户已经确认：“现在文件导出没问题了。”

### 4.2 TestGradient 程序已重构增强

当前测试程序 `opengldp_benchmark` 已经承担以下职责：

- 解析 benchmark 验证
- 与 VTK 结果对照
- 计时统计
- 批量实验
- CSV 导出

其设计目标是让 GUI 与测试程序尽量复用同一条加载和计算主线，避免出现“两套逻辑算出两套结果”的问题。

## 5. 关于真实数据与解析 benchmark 的当前结论

这是当前最重要的判断，后续新对话不要再从头摇摆。

### 5.1 能确认的

1. 解析场 benchmark 在多个数据集上整体是通的。
2. 这说明当前几何邻域构造、局部坐标、WLS/AWLS 求解主链路并没有整体失效。
3. 因此不能把当前版本简单判定为“梯度模块完全不可用”。

### 5.2 还不能确认的

1. 真实工程场与 `vtkGradientFilter` / ParaView 的结果差异，不能直接说明谁一定对谁一定错。
2. 真实工程场没有解析真值时，和 VTK/ParaView 的比较只能算“工程对照”，不能算严格真值验证。
3. 即使解析 benchmark 很好，也不能直接写成“模块已经成熟到全面适用于所有真实非结构网格场数据”。

更稳妥的论文口径应该是：

- 解析 benchmark 证明算法在受控场上的正确性与稳定性。
- 真实数据对比用于说明工程表现、适用范围和局限性。

## 6. 当前最关键的现实问题

当前最需要继续定位的，不是导出问题，而是：

- 为什么真实数据，尤其是 `notch_stress` 和 `ShipHull_0`，与 ParaView/VTK 对照时视觉结果明显不好。

这个问题现在不能简单归因成“算法完全错了”，更可能是以下几类因素叠加：

1. 真实场本身不平滑、有噪声、或是有限元后处理场。
2. 数据可能是壳/表面/薄层结构，对法向分量非常敏感。
3. ParaView/VTK 与当前实现并不一定在同一个口径下比较。
4. GUI 显示方式可能与 ParaView 不一致。

## 7. `ShipHull_0` 的已确认结论

这一点非常关键，后续新对话必须先知道。

### 7.1 数据本身是什么

`Data/ShipHull_0.vtk` 不是体网格，而是一个比较粗的三维表面网格。

已确认：

- `POINTS 405`
- `CELLS 268`
- `CELL_TYPES` 基本为 `9`，即 `vtkQuad`
- `POINT_DATA` 下有向量数组：
  - `RF`
  - `RM`
  - `U`
  - `UR`
- `CELL_DATA` 下有标量数组：
  - `S_Mises`

这意味着 `ShipHull_0` 更接近“嵌入三维空间的二维壳/表面网格”，而不是普通三维体网格。

### 7.2 当前 GUI 与 ParaView 的对比很可能不是同一量

代码里已确认：

- GUI 渲染使用 `SelectColorArray(...)`
- 然后调用 `ColorByArrayComponent(...)`

也就是说，当前系统 GUI 是“按某个分量上色”，不是“按模长上色”。

而用户给出的 ParaView 截图中，之前明确看到过 `Gradient` 的 `Magnitude` 选项。

因此非常可能出现如下不公平对比：

- ParaView：显示 `|grad|`
- 系统 GUI：显示 `grad_x` 或当前选中的某一个分量

这种情况下，两张图看起来差很多，并不能直接说明系统梯度计算更差。

### 7.3 对 `ShipHull_0` 更合理的现阶段判断

当前更合理的结论是：

1. `ShipHull_0` 是一个更难的数据类型，因为它是粗糙表面网格。
2. 当前 AWLS 已经有局部维数自适应，不是简单地把它硬算成纯三维体梯度。
3. 但 GUI 与 ParaView 的显示口径尚未统一，这会放大视觉差异。
4. 在没有统一“点/单元关联、分量/模长、色标范围”的前提下，不能只靠截图判断谁更好。

## 8. `notch_stress` 的已确认现状

`notch_stress` 目前仍然是重要困难样例。

已确认现象：

1. 解析 benchmark 在该类数据上并非全面崩溃，说明主链路可运行。
2. 但真实场与 VTK/ParaView 对照时，结果可能明显不好看。
3. 这更像是“真实场稳定性 / 表面或壳结构处理 / 显示口径”问题，而不是单纯的语法或导出问题。

当前对 `notch_stress` 更稳妥的怀疑方向：

1. 真实应力场本身不够平滑。
2. 壳/表面/局部奇异区域导致法向分量放大。
3. 当前支撑构造、维数判断或正则化在这类场上还不够稳。
4. VTK 的梯度定义和当前 AWLS 的定义并不完全一致。

## 9. 测试与对照时必须统一的口径

以后新对话接手时，如果要比较系统与 ParaView/VTK，必须先统一以下条件：

1. 同一个输入字段。
2. 同一个关联：
   - `POINT`
   - 或 `CELL`
3. 同一种显示方式：
   - 某一固定分量
   - 或模长
4. 同一个色标范围。
5. 同样的是否包含边界、异常值和 `NaN` 处理方式。

如果这些条件不统一，截图对比的结论不可靠。

## 10. 当前最推荐的后续推进顺序

如果后续 AI 要继续接手，建议优先按下面顺序推进：

1. 先把 GUI 增加“模长显示”能力。
   - 3 分量梯度显示向量模长
   - 9 分量梯度显示 Frobenius norm 或明确说明显示规则
2. 重新对 `ShipHull_0` 做真正公平的对照：
   - 同一字段
   - 同一关联
   - 同一分量或同一模长
   - 同一色标
3. 对 `notch_stress` 补诊断输出，而不是继续只看误差总指标：
   - `dimTag` 分布
   - `quality` 的最小/最大/均值
   - 邻居数最小/最大/均值
   - 被判成 2D/3D 的比例
   - 法向分量是否异常放大
4. 如果继续确认是表面/壳数据问题，再考虑进一步增强：
   - 对 `dimTag == 2` 的样本加强切向约束
   - 在局部二维平面中拟合后再映回三维
   - 增加专门的 surface-aware 梯度显示或计算选项

## 11. 当前构建与环境状态

已确认：

1. `opengldp_benchmark` 可以成功编译。
2. 核心算法代码并没有出现明显编译级崩坏。
3. GUI 的完整构建曾在沙箱中因为 `moc.exe` 子进程权限问题失败。
4. 这更像环境/权限问题，不像当前代码语义错误。

典型报错是：

- `libuv process spawn failed: operation not permitted`

因此：

- benchmark 与核心模块基本可编译、可运行；
- GUI 构建问题需要结合本机环境权限看，不宜误判为算法代码错误。
